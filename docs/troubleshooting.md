# Troubleshooting

## Initialization and configuration

**`aiyoke.yaml was not found`** — Confirm `--root`, then run `aiyoke init`.
Running from a monorepo parent without `--root` intentionally does not search
arbitrary descendants.

**Configuration already exists** — Use `aiyoke config` for edits. `init --force`
replaces the canonical source and should be used only after reviewing the current
file or creating a backup.

**Unknown extension / missing requirement / conflict** — Run `aiyoke list`, use
exact registered IDs, and select required language/framework dependencies. The
registry rejects duplicates, missing requirements, cycles, and declared conflicts
before rendering.

**Interactive mode fails in automation** — `config --interactive` requires both
input and output TTYs. Use deterministic flags and `--json` in CI.

## Planning and generation

**`plan` exits 1** — Inspect `conflict` operations. Common causes are two sources
owning one path, malformed/duplicate managed markers, or a user-owned file at a
generated path. `apply` will not bypass ownership conflicts.

**Stale plan/source conflict** — Re-run `plan` after any canonical configuration
or workspace change. Aiyoke fingerprints operations and rechecks source state to
avoid applying an obsolete decision.

**Symlink or non-directory rejection** — Generated paths must remain below a
canonical real workspace directory. Replace the ancestor symlink with a real
directory or choose another output path. This containment rule is not
configurable.

**`check` reports drift** — Review the canonical source and `plan`. If generation
is expected, run `apply`; otherwise restore the reviewed generated bytes. Do not
edit the lock digest to hide drift.

## Schemas and backups

**Older schema rejected** — Run `migrate --dry-run` first. Normal commands never
silently migrate.

**Downgrade refused with consent** — The current document contains state that the
older schema cannot represent. Remove or transform that state explicitly; Aiyoke
will not perform a lossy downgrade.

**Rollback source changed** — Another process edited `aiyoke.yaml` while recovery
was prepared. Reopen/review both sources and retry. The safety check intentionally
prevents replacement.

## Providers and runtime templates

**Missing provider credential** — Aiyoke generation uses only environment-variable
names. Inject the actual value into the generated application's secret resolver;
do not put it in `aiyoke.yaml` or generated files.

**Live smoke is skipped** — Set `AIYOKE_LIVE_PROVIDER_TESTS=1` and provide an
ignored local `.env` containing `OPENROUTER_API_KEY`. The default test suite never
loads credentials or performs network calls.

**Rust provider adapter has no HTTP client** — This is deliberate. Rust's standard
library has no HTTPS client, so the generated stable transport port is the
integration boundary for the consuming application's selected TLS/HTTP crate.

## Extensions

**Consent requested again** — Approval is bound to the canonical manifest digest.
Any descriptor, package, digest, entrypoint, version, or publisher change requires
new explicit consent.

**Signed package rejected before import** — Check strict manifest shape, package
digest, Ed25519 key ID/public key, key/content/manifest revocations, symlinks,
resource limits, and consent digest. Aiyoke intentionally returns bounded generic
messages instead of hostile package exceptions.

**Isolated renderer fails** — Confirm extension kind matches the target/runtime
invocation, output paths and managed markers are valid, output stays within byte
and artifact limits, and the deadline is sufficient. Child stdout is not the
protocol. For OS-level filesystem/network denial, add a container or VM sandbox.

## Development and release checks

Use Node 22 or 24 and pnpm 11.7.0. Start with `pnpm install --frozen-lockfile`,
then run `pnpm check`. Native runtime/framework gates additionally require the
versions configured in `.github/workflows/ci.yml`.

If local `pnpm audit` reports `UNABLE_TO_VERIFY_LEAF_SIGNATURE` inside an embedded
Node runtime, use a normally installed Node distribution or Node's system CA mode;
do not disable TLS verification.
